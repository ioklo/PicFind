# 아키텍처

## 목표

로컬 우선 방식의 사진 검색 도구를 제공한다.

- 사용자가 관리하는 사진 폴더를 인덱싱한다.
- 각 이미지의 짧은 설명을 생성해 저장한다.
- 문장 검색으로 의미상 비슷한 사진을 찾는다.
- 검색 결과를 빠르게 확인할 수 있는 로컬 UI를 제공한다.
- 아이폰 사진 라이브러리에서 자주 쓰이는 HEIC/HEIF도 읽을 수 있어야 한다.

## MVP 구성 요소

### CLI

인덱싱과 점검은 명령줄 인터페이스로 제공한다. 자동화에 유리하고 대량 처리에 적합하다.

### 웹 UI

Streamlit 기반 UI로 검색 질의, 상위 결과, 썸네일, 캡션을 확인한다. 인덱싱 자체는 CLI에서 수행하고 UI는 조회 중심으로 사용한다.

### 설명 생성기

BLIP가 각 이미지에 대한 짧은 캡션을 만든다. 생성된 설명은 사람이 확인하거나 나중에 필터링에 쓸 수 있다.

### 임베딩 생성기

CLIP은 이미지와 사용자 문장을 같은 벡터 공간으로 보낸다. 검색은 정규화된 벡터 간 코사인 유사도로 계산한다.

### 저장소

SQLite에 다음 정보를 저장한다.

- 파일 경로와 메타데이터
- 생성된 설명
- 직렬화된 임베딩 벡터

### 이미지 로더

기본 이미지는 Pillow로 읽고, HEIC/HEIF는 `pillow-heif` 플러그인을 등록해서 같이 처리한다.

### 모델 로더

모델 로딩은 `safetensors` 형식을 우선 시도한다. 이 형식이 없거나 현재 `torch` 버전이 너무 낮으면, 사용자가 `torch>=2.6` 으로 올릴 수 있도록 명확한 오류를 제공한다.

### 검색

검색 시 사용자 문장을 CLIP 텍스트 인코더로 임베딩하고, 저장된 이미지 임베딩과 비교해서 상위 결과를 돌려준다.

## 데이터 흐름

1. 소스 디렉터리 아래의 이미지 파일을 찾는다.
2. 파일 크기와 수정 시각을 읽는다.
3. 변경되지 않은 파일은 건너뛴다.
4. 새 파일이나 변경된 파일에 대해 캡션과 임베딩을 생성한다.
5. 결과를 SQLite에 upsert 한다.
6. 검색 시 질의를 임베딩하고 코사인 유사도로 정렬한다.
7. 웹 UI는 상위 결과를 읽어 썸네일과 메타데이터를 보여준다.

## MVP 한계

- 검색은 ANN 인덱스가 아니라 저장된 벡터 전체를 순회하는 방식이다.
- 캡션 품질은 BLIP 모델 성능에 영향을 받는다.
- GPU는 선택 사항이지만, 인덱싱 속도에는 큰 차이가 난다.
- 기본 CLIP 모델은 영어 중심이라 한국어 검색 품질이 균일하지 않을 수 있다.
- 작은 물체나 미세한 상태를 묻는 질의는 정확도가 떨어질 수 있다.
